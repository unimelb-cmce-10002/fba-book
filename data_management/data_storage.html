7  Storing and Using Existing Data

Learning Goals

By the end of this lecture, you should be able to:

  • Identify common data storage formats (CSV, JSON, SQL, Parquet) and explain when each is appropriate for business use.
  • Load, explore, and manipulate datasets stored in flat files and nested JSON using R.
  • Use unnesting and reshaping techniques to convert nested or redundant data structures into tidy formats.
  • Read and query data stored in DuckDB and Parquet formats using dbplyr and arrow.
  • Evaluate trade-offs between different storage formats in terms of scalability, structure, and analytical convenience.
  • Apply best practices for storing, documenting, and sharing data within an organization.

7.1 The Business Challenge

Online marketplaces like eBay host millions of auctions each year, connecting buyers and sellers across countless product categories. Behind the scenes, this generates complex, nested, and high-volume data. For a business analyst, turning this raw data into useful insights‚Äîand storing it in a way that‚Äôs efficient, accessible, and reusable‚Äîis a foundational challenge.

You‚Äôve just joined the data team at a fast-growing online marketplace. Right now, every team ‚Äî from marketing to data engineering and customer support ‚Äî is handling auction data in different ways: some use Excel sheets, others parse logs from an API that stores information in a format called JSON, and a few engineers keep things in a different format known as Parquet (but no one else knows how to read them!).

This chaos is slowing everyone down. Your first task as a new analyst is to propose a better data storage solution that balances usability, scalability, and efficiency for teams across the business.

But how do you choose the right format? Should you push everyone to use Excel spreadsheets? Convince IT to move everything into a database? Or are columnar formats like Parquet worth the learning curve?

Before you decide, you‚Äôll need to understand the strengths and trade-offs of each major data storage format. That is what we will cover in this lecture

About the Data

To guide your decision, we‚Äôll work with a sample dataset of online auctions. Each auction record includes:

  • Item metadata: ID, title, category, seller info
  • Seller details: user ID, rating
  • Bidding history: a nested list of bids, each with a bidder ID, amount, and timestamp

We have stored and structured the dataset in multiple formats ‚Äî Excel, flat files (CSV), JSON, DuckDB, and Parquet‚Äîso you can compare how different formats impact usability and performance. Along the way, you‚Äôll learn not just how to load data, but how to think about storing and transforming it for long-term use in a business setting.

7.2.3 Issues with Spreadsheets

They‚Äôre widely used in business and education, and offer an approachable interface for organizing tables, applying formulas, and making quick charts. This makes spreadsheets a great entry point into the world of data‚Äîbut also a format with serious limitations when it comes to analytics at scale.

Spreadsheets don‚Äôt record your steps, meaning you can‚Äôt always retrace how a figure was calculated. There‚Äôs no built-in version history (at least not one that‚Äôs easy to audit), and formulas often vary invisibly from cell to cell‚Äîespecially if you‚Äôve copied or dragged them across a range. This makes it difficult to ensure consistency and nearly impossible to reproduce or validate your results.

Even small errors‚Äîlike accidentally referencing the wrong column‚Äîcan go unnoticed. A single cell with a broken formula can distort an entire analysis. These issues are especially problematic when figures or charts are being used for decisions or presentations.

To make matters worse, analysts often inherit spreadsheets from others‚Äîcolleagues, clients, or external partners. These files might contain:

  • inconsistent naming conventions
  • manual totals
  • hidden formatting or comments
  • merged cells
  • duplicated values across tabs

Despite these flaws, spreadsheets remain a powerful tool for quick exploration and lightweight collaboration. They‚Äôre useful for small teams, early prototyping, or downloading data from platforms like Shopify, Facebook Ads, or Google Analytics. But when your work moves toward reproducibility, scalability, or automation, it‚Äôs time to adopt formats designed for analysis.

7.3 Flat Files (CSV, TSV and other delimited files)

when it comes to storing and sharing structured data reliably, especially in a way that can be reproduced by code, an alternative format often works better: flat files.

Flat files look similar to spreadsheets on the surface‚Äîjust rows and columns of data‚Äîbut under the hood they behave very differently.

A flat file is stored as plain text. Each line represents a row of data, and the values are separated by a special character like a comma or tab. There are no formulas, no styling, no hidden cells‚Äîjust the raw information.

One of the most common flat file types is the CSV, which stands for Comma-Separated Values. You might have encountered files with a .csv extension when exporting data from online dashboards or business tools.

Here‚Äôs what a CSV might look like when opened in a text editor:

item_id,bidder_id,amount,time
a001,u07,45.00,2023-08-21 10:13
a001,u12,47.50,2023-08-21 10:30

This format is:

  • Easy to inspect and edit
  • Supported by nearly every tool
  • A common default for data exports

7.3.2 Why are the same values repeated?

When you look at the bids for one auction, you might notice that the item title, category, and seller name appear again and again in every row. That might seem strange‚Äîwhy repeat the same information?

This happens because of the way flat files are designed. In a flat file, each row represents one observation‚Äîin this case, one bid. That means every row needs to include not just the bid amount and time, but also the details of the auction it belongs to. Even if those auction details don‚Äôt change, they‚Äôre still copied into every row.

This approach is useful for analysis, but it can also lead to large files and repeated information. In more advanced systems, like databases, we might store the auction information just once and link each bid to it. But flat files keep everything together in one place, which makes them easy to share and work with.

Later in this chapter, we‚Äôll see how other formats (like JSON and SQL) handle this differently‚Äîand how that affects the way we store and analyze data.

7.4 JSON Files: When rectangular data might not be enough

Flat files are great when every observation fits neatly into one row. But what if each item in your dataset comes with its own mini-dataset inside‚Äîlike multiple bids per auction?

Trying to represent that in a flat file leads to a lot of repetition. Every bid needs to repeat the same auction information‚Äîitem title, seller name, category‚Äîeven if it doesn‚Äôt change. This design follows the tidy data principle, where each row is a single observation and each column is a variable. It‚Äôs ideal for analysis, but not always the most efficient way to store structured or repeated data.

To better handle these cases, we can use nested data formats‚Äîand one of the most popular is JSON.

JSON (JavaScript Object Notation) is a format designed to store structured data. It‚Äôs especially useful when one record contains multiple layers of information‚Äîfor example, an auction that has several bids attached to it.

Why JSON?

  • Captures nested structures, such as multiple bids grouped within a single auction
  • Common in APIs, log files, and web applications
  • More compact and organized for data with repeating elements

Unlike flat files, JSON allows us to store all related data in one object, without repeating fields. It‚Äôs human-readable, flexible, and widely used in real-world business data systems.

JSON (short for JavaScript Object Notation) stores data as a set of key-value pairs. Think of it like a list of labeled boxes‚Äîeach label is the name of a variable, and inside each box is the value.

In the context of our auction dataset, one auction might look like this in JSON format:

{
  "item_id": "a001",
  "title": "Vintage Coffee Table",
  "category": "Furniture",
  "seller": {
    "user_id": "seller42",
    "rating": 4.8
  },
  "bids": [
    { "bidder_id": "user123", "amount": 120, "time": "2023-01-01 10:05" },
    { "bidder_id": "user456", "amount": 130, "time": "2023-01-01 11:20" }
  ]
}

Let‚Äôs break it down:

  • The top-level keys (item_id, title, category) describe the auction itself.
  • The seller key contains a nested object with seller details.
  • The bids key holds a list of bid objects, each with its own bidder, amount, and time.

This structure keeps all related information grouped together‚Äîno need to repeat the title or seller info for every bid like we did in the CSV file. That‚Äôs (one of) the power(s) of JSON: it stores complex relationships without flattening the data.

ADD callout: JSON does not have to be nested!

7.4.1 Working with Nested JSON Auction Data in R

In this section, we‚Äôll learn how to work with nested JSON data. Each auction record includes top-level information (like item title and category), seller information, and a nested list of bids placed by users. This structure is very common in the real world‚Äîespecially when dealing with data from APIs or web applications.

Our goal is to explore how to load and manipulate this nested structure using tidyverse tools. We‚Äôll move step by step, from raw JSON to a tidy tibble.

7.4.2 Step 1: Load the JSON File

We‚Äôll start by reading in the data using the read_json() function from the jsonlite package. This preserves the nested list structure exactly as it appears in the JSON file.

library(jsonlite)

The object auctions_nested is a list of auction records. Each element in the list is a nested list with keys like item_id, title, seller, and bids.

Each row of auctions_tbl now contains one full auction record.

7.4.5 Rectangling non-rectangular data

you‚Äôll learn the art of data rectangling: taking data that is fundamentally hierarchical, or tree-like, and converting it into a rectangular data frame made up of rows and columns. This is important because hierarchical data is surprisingly common, especially when working with data that comes from the web.

CALLOUT ON R lists

7.4.6 Step 4: Unnest the Top-Level Fields

We can begin flattening the nested structure one layer at a time using unnest_wider():

auctions_tbl <- auctions_tbl %>%
  unnest_wider(record)

Now we have columns like item_id, title, category, seller, and bids. Both seller and bids are still lists.

When each row has the same number of elements with the same names, like XXX, it‚Äôs natural to put each component into its own column with unnest_wider():

auctions_tbl <- auctions_tbl %>%
  unnest_wider(seller, names_sep = "_")

Now seller information like user_id and rating is available as separate columns.

7.4.7 Step 5: Flattening the Bid Lists (Wide Format)

What about the bids column? Each auction has its own list of bids, and each bid is a small list too. First, we can break it out into wide format:

auctions_tbl_wide <- auctions_tbl %>%
  unnest_wider(bids, names_sep = "_")

This gives us one column for each bid: bids_1, bids_2, etc.‚Äîbut these are still lists. We can unnest them as well:

auctions_tbl_wide <- auctions_tbl_wide %>%
  unnest_wider(bids_1, names_sep = "_", names_repair = "unique") %>%
  unnest_wider(bids_2, names_sep = "_", names_repair = "unique")

This gets tedious if there are many bids. So let‚Äôs automate it.

7.4.8 Step 6: Automate the Unnesting of All Bid Columns

Use this if you don‚Äôt know how many bids there will be ahead of time:

library(purrr)

bid_cols <- auctions_tbl_wide %>%
  select(starts_with("bids_")) %>%
  select(where(is.list)) %>%
  names()

auctions_tbl_wide <- reduce(
  bid_cols,
  .init = auctions_tbl_wide,
  .f = function(df, col) {
    unnest_wider(df, !!sym(col), names_sep = "_", names_repair = "unique")
  }
)

7.4.9 Step 7: Unnesting to Long Format

Discuss why the wider might not be what we wanted ‚Ä¶

Instead of going wider, we can go longer. This means one row per bid:

bids_long <- auctions_tbl %>%
  unnest_longer(bids) %>%
  unnest_wider(bids)

Now we have a tidy table where each row is a bid, including the item ID and other metadata. This is often easier to work with for analysis and looks very much like the CSV format!.

7.4.10 Bonus: Nesting Back Again

Let‚Äôs reverse the process and group the bids back into a nested column:

bids_renested <- bids_long %>%
  select(item_id, bidder_id, amount, time) %>%
  group_by(item_id) %>%
  summarise(bids = list(cur_data()), .groups = "drop")

7.5.1 Why databases?

A huge amount of data lives in databases, so it‚Äôs essential that you know how to access it.

  • Efficient for large structured data
  • Use SQL to join/filter/aggregate

TIP BOX

Databases are run by database management systems (DBMS‚Äôs for short), which come in three basic forms:

  • Client-server DBMS‚Äôs run on a powerful central server, which you connect to from your computer (the client). They are great for sharing data with multiple people in an organization. Popular client-server DBMS‚Äôs include PostgreSQL, MariaDB, SQL Server, and Oracle.
  • Cloud DBMS‚Äôs, like Snowflake, Amazon‚Äôs RedShift, and Google‚Äôs BigQuery, are similar to client server DBMS‚Äôs, but they run in the cloud. This means that they can easily handle extremely large datasets and can automatically provide more compute resources as needed.
  • In-process DBMS‚Äôs, like SQLite or duckdb, run entirely on your computer. They‚Äôre great for working with large datasets where you‚Äôre the primary user.

7.5.2 Connecting to a database

To connect to the database from R, you‚Äôll use a pair of packages:

You‚Äôll always use DBI (database interface) because it provides a set of generic functions that connect to the database, upload data, run SQL queries, etc.

You‚Äôll also use a package tailored for the DBMS you‚Äôre connecting to. This package translates the generic DBI commands into the specifics needed for a given DBMS.

we‚Äôll instead use an in-process DBMS that lives entirely in an R package: duckdb. Thanks to the magic of DBI, the only difference between using duckdb and any other DBMS is how you‚Äôll connect to the database. This makes it great to teach with because you can easily run this code as well as easily take what you learn and apply it elsewhere.

Connecting to duckdb is particularly simple because the defaults create a temporary database that is deleted when you quit R. That‚Äôs great for learning because it guarantees that you‚Äôll start from a clean slate every time you restart R:

con <- DBI::dbConnect(duckdb::duckdb())

duckdb is a high-performance database that‚Äôs designed very much for the needs of a data scientist. We use it here because it‚Äôs very easy to get started with, but it‚Äôs also capable of handling gigabytes of data with great speed.

TIP If you want to use duckdb for a real data analysis project, you‚Äôll also need to supply the dbdir argument to make a persistent database and tell duckdb where to save it

7.5.4 dbplyr

Now that we‚Äôve connected to a database and loaded up some data, we can start to learn about dbplyr. dbplyr is a dplyr backend, which means that you keep writing dplyr code but the backend executes it differently. In this, dbplyr translates to SQL;

To use dbplyr, you must first use tbl() to create an object that represents a database table:

diamonds_db <- tbl(con, "diamonds")
diamonds_db

This object is lazy; when you use dplyr verbs on it, dplyr doesn‚Äôt do any work: it just records the sequence of operations that you want to perform and only performs them when needed. For example, take the following pipeline:

big_diamonds_db <- diamonds_db |> 
  filter(price > 15000) |> 
  select(carat:clarity, price)

You can tell this object represents a database query because it prints the DBMS name at the top, and while it tells you the number of columns, it typically doesn‚Äôt know the number of rows. This is because finding the total number of rows usually requires executing the complete query, something we‚Äôre trying to avoid.

To get all the data back into R, you call collect(). Behind the scenes, this generates the SQL, calls dbGetQuery() to get the data, then turns the result into a tibble:

library(dplyr)
bids_db <- tbl(con, "bids") %>% filter(category == "Books") %>% collect()

How did that work? You can see the SQL code generated by the dplyr function show_query(). If you know dplyr, this is a great way to learn SQL! Write some dplyr code, get dbplyr to translate it to SQL, and then try to figure out how the two languages match up.

Typically, you‚Äôll use dbplyr to select the data you want from the database, performing basic filtering and aggregation using the translations described below. Then, once you‚Äôre ready to analyse the data with functions that are unique to R, you‚Äôll collect() the data to get an in-memory tibble, and continue your work with pure R code.

Opening a dataset Let‚Äôs start by taking a look at the data. At 9 GB, this file is large enough that we probably don‚Äôt want to load the whole thing into memory. 7.6.1.1 The parquet format

To make this data easier to work with, let‚Äôs switch to the parquet file format and split it up into multiple files. The following sections will first introduce you to parquet and partitioning, and then apply what we learned to the Seattle library data.

Advantages of parquet Like CSV, parquet is used for rectangular data, but instead of being a text format that you can read with any file editor, it‚Äôs a custom binary format designed specifically for the needs of big data. This means that:

Parquet files are usually smaller than the equivalent CSV file. Parquet relies on efficient encodings to keep file size down, and supports file compression. This helps make parquet files fast because there‚Äôs less data to move from disk to memory.

Parquet files have a rich type system. As we talked about in Section 7.3, a CSV file does not provide any information about column types. For example, a CSV reader has to guess whether ‚Äú08-10-2022‚Äù should be parsed as a string or a date. In contrast, parquet files store data in a way that records the type along with the data.

Parquet files are ‚Äúcolumn-oriented‚Äù. This means that they‚Äôre organized column-by-column, much like R‚Äôs data frame. This typically leads to better performance for data analysis tasks compared to CSV files, which are organized row-by-row.

Parquet files are ‚Äúchunked‚Äù, which makes it possible to work on different parts of the file at the same time, and, if you‚Äôre lucky, to skip some chunks altogether.

There‚Äôs one primary disadvantage to parquet files: they are no longer ‚Äúhuman readable‚Äù, i.e. if you look at a parquet file using readr::read_file(), you‚Äôll just see a bunch of gibberish.

22.4.2 Partitioning As datasets get larger and larger, storing all the data in a single file gets increasingly painful and it‚Äôs often useful to split large datasets across many files. When this structuring is done intelligently, this strategy can lead to significant improvements in performance because many analyses will only require a subset of the files.

There are no hard and fast rules about how to partition your dataset: the results will depend on your data, access patterns, and the systems that read the data. You‚Äôre likely to need to do some experimentation before you find the ideal partitioning for your situation. As a rough guide, arrow suggests that you avoid files smaller than 20MB and larger than 2GB and avoid partitions that produce more than 10,000 files. You should also try to partition by variables that you filter by; as you‚Äôll see shortly, that allows arrow to skip a lot of work by reading only the relevant files.

Let‚Äôs take a look at what we just produced:

Our single 9GB CSV file has been rewritten into 18 parquet files. The file names use a ‚Äúself-describing‚Äù convention used by the Apache Hive project. Hive-style partitions name folders with a ‚Äúkey=value‚Äù convention, so as you might guess, the CheckoutYear=2005 directory contains all the data where CheckoutYear is 2005. Each file is between 100 and 300 MB and the total size is now around 4 GB, a little over half the size of the original CSV file. This is as we expect since parquet is a much more efficient format.

Using dplyr with arrow Now we‚Äôve created these parquet files, we‚Äôll need to read them in again. We use open_dataset() again, but this time we give it a directory:

seattle_pq <- open_dataset(pq_path)

Now we can write our dplyr pipeline. For example, we could count the total number of books checked out in each month for the last five years:

query <- seattle_pq |> filter(CheckoutYear >= 2018, MaterialType == ‚ÄúBOOK‚Äù) |> group_by(CheckoutYear, CheckoutMonth) |> summarize(TotalCheckouts = sum(Checkouts)) |> arrange(CheckoutYear, CheckoutMonth)

Performance Let‚Äôs take a quick look at the performance impact of switching from CSV to parquet. First, let‚Äôs time how long it takes to calculate the number of books checked out in each month of 2021, when the data is stored as a single large csv:

seattle_csv |> 
  filter(CheckoutYear == 2021, MaterialType == "BOOK") |>
  group_by(CheckoutMonth) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(desc(CheckoutMonth)) |>
  collect() |> 
  system.time()
#>    user  system elapsed 
#>  11.951   1.297  11.387

Now let‚Äôs use our new version of the dataset in which the Seattle library checkout data has been partitioned into 18 smaller parquet files:

seattle_pq |> 
  filter(CheckoutYear == 2021, MaterialType == "BOOK") |>
  group_by(CheckoutMonth) |>
  summarize(TotalCheckouts = sum(Checkouts)) |>
  arrange(desc(CheckoutMonth)) |>
  collect() |> 
  system.time()
#>    user  system elapsed 
#>   0.263   0.058   0.063

The ~100x speedup in performance is attributable to two factors: the multi-file partitioning, and the format of individual files:

Partitioning improves performance because this query uses CheckoutYear == 2021 to filter the data, and arrow is smart enough to recognize that it only needs to read 1 of the 18 parquet files. The parquet format improves performance by storing data in a binary format that can be read more directly into memory. The column-wise format and rich metadata means that arrow only needs to read the four columns actually used in the query (CheckoutYear, MaterialType, CheckoutMonth, and Checkouts). This massive difference in performance is why it pays off to convert large CSVs to parquet!
